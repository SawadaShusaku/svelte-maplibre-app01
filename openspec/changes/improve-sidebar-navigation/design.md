## Context

現在のサイドバーは `SettingsSidebar.svelte` に集約されており、実質的な操作はマーカーデザイン設定のみで、残りのメニュー項目はプレースホルダーのボタンになっている。この構成だと、ユーザーにとって「設定を変える場所」と「アプリやデータの補足情報に進む場所」の境界が曖昧で、区データの拡充後に必要になる案内導線の優先順位も表現できない。

一方で、このアプリは地図 UI だけでなく、自治体オープンデータの出所、更新状況、使い方を補足する案内導線の重要度が高い。特にデータ品質への信頼を支えるため、`データについて` や `更新情報` のような項目は、単なる将来の飾りではなく、情報設計上の優先項目として扱う必要がある。

## Goals / Non-Goals

**Goals:**
- サイドバーを `表示設定` と `情報・ヘルプ` に分け、役割の異なる項目を明確にする
- 現在の主要設定であるマーカーデザインを、初見でも見つけやすく操作しやすい形に整える
- 情報系メニューを優先順位つきで整理し、今後 SvelteKit の別URLページへ接続しやすい表現にする
- データ品質や更新背景に関する導線を上位に置ける構造を用意する

**Non-Goals:**
- `使い方` や `データについて` の本文ページそのものをこの change で完成させること
- 区の掲載状況一覧をサイドバーに追加すること
- ヘッダー側の検索、区選択、カテゴリ選択の位置や役割を変更すること
- サイドバーを別 UI パターンのモーダルやフルページナビゲーションへ置き換えること

## Decisions

### 1. サイドバーをセクション化し、設定系と案内系を分離する
`SettingsSidebar.svelte` の中で、少なくとも `表示設定` と `情報・ヘルプ` の2セクションを明示する。これにより、ユーザーは「ここでは見た目を調整できる」「ここから補足情報へ進める」を一目で判断できる。

代替案として単一リストのまま見出しだけ追加する案もあるが、現在の UI ではメニュー項目がプレースホルダーに見えやすく、役割分離が弱い。セクション単位で情報密度を分けたほうが、今後項目が増えても破綻しにくい。

### 2. マーカーデザインは主要設定として常時見える形を優先する
現在の折りたたみ UI は、サイドバーを開いても実質的な設定がすぐ見えない。マーカーデザインは現状の主機能なので、初期展開状態にするか、カード風のまとまりとして常時表示する方針を採る。

代替案としてすべてを折りたたみ可能にする方法もあるが、設定項目数がまだ少ない現段階では、情報を隠す効果より発見性の低下の方が大きい。

### 3. 情報系メニューは設定と異なる「リンク行」表現に統一する
情報系項目は、単なるプレーンボタンではなく、将来の別URL遷移を前提にしたリンク行として設計する。各行は `label`、任意の `description`、任意の `href`、状態（利用可能 / 準備中）を持つ構造にする。

これにより、実装初期は `href` 未設定の項目を `準備中` として扱い、後から SvelteKit ルートに接続してもコンポーネント構造を崩さずに拡張できる。

### 4. 情報系メニューの優先順位を固定し、データ品質に関わる項目を上位に置く
初期構成は次の順序を前提とする。

1. `使い方`
2. `データについて`
3. `更新情報`
4. `プライバシーポリシー`

`掲載区一覧` は追加しない。掲載範囲は地図・区選択・検索結果から把握でき、サイドバーの情報密度を上げる割に導線価値が低いためである。

### 5. メニュー定義は配列データに寄せ、将来のページ追加に備える
`SettingsSidebar.svelte` 内にベタ書きした複数ボタンのまま増やすのではなく、少なくとも情報系項目は配列定義から描画する。これにより、後続 change で `/help`, `/about-data`, `/updates`, `/privacy` のような別ルートを用意したときに、文言や順番、公開状態を一箇所で管理できる。

## Risks / Trade-offs

- サイドバーに情報項目を増やしすぎると、地図の補助 UI ではなく主ナビゲーションに見えてしまう
  → 情報系は4件程度に絞り、設定系と視覚的に分離する
- `準備中` 項目を見せると、押したときの期待を裏切る可能性がある
  → 遷移不可の状態表示を明示し、利用可能になるまでは通常リンクと見た目を分ける
- 設定 UI を常時展開すると縦長になる
  → 主設定を1まとまりに留め、情報項目は簡潔な行 UI にして密度を抑える

## Migration Plan

1. サイドバーのセクション構成を更新し、既存のマーカーデザイン設定を `表示設定` 配下へ移す
2. 情報系項目をリンク行コンポーネントまたは配列描画に置き換え、優先順位順に表示する
3. `href` を持たない項目を将来の遷移先追加に備えた状態付きメニューとして扱う
4. 将来の change で各情報ページのルートを追加する際は、同じメニュー定義へ遷移先を接続する

ロールバック時は、リンク行データ構造を外して既存のボタン列へ戻せばよい。データ層やルーティング層への破壊的変更は想定しない。

## Open Questions

- `使い方` と `データについて` を最初の実装で実ページ化するか、まずは `準備中` 表示に留めるか
- 情報系項目の遷移は内部ルートに統一するか、将来一部を外部公式ページへ直接リンクすることを許容するか
- モバイルで情報系項目が増えた場合、現状のサイドパネルのままで十分か、別 change でフルハイト化を検討するか
