## Context

現在の地図画面は `src/routes/+page.svelte` に施設取得、マーカー描画、popup 制御、検索結果反映が集中している。施設数が 23 区分まで増えたことで、DOM ベースの `Marker` を全件描画する構成ではズームアウト時のレイアウト・ペイント負荷が高くなっている。一方で、現行のマーカーデザイン自体は確定仕様であり、視覚表現は変更できない。

また、popup は施設ループ内でデスクトップ用 `Popup` を条件付き描画し、モバイルでは単一ボトムシートを出す構成になっている。実際に表示される popup は 1 件でも、選択状態と表示責務が分散しており、描画戦略の変更と合わせて整理したい。

GeoJSON スキーマや SQLite 生成フローは変えず、GeoJSON を単一の正本とする要件も維持する。

## Goals / Non-Goals

**Goals:**
- 低ズーム時の地図操作を滑らかにするため、施設の個別描画数を抑える
- 現行のマーカーデザインを変更せずに描画負荷を下げる
- popup / ボトムシートの表示対象を単一の選択施設 state に集約する
- `+page.svelte` からマーカー描画最適化の責務を分離し、描画戦略を差し替えやすくする
- フィルタ変更時に施設データと描画データを必要以上に再生成しないようにする

**Non-Goals:**
- マーカー形状、配色、グラデーション、カテゴリ表現の変更
- GeoJSON スキーマの変更や DB マイグレーション仕様の変更
- 検索仕様、ルーティング仕様、popup コンテンツ内容の変更
- 低ズーム時の区サマリー表示の見た目を、既存施設マーカーと同一にすること

## Decisions

### 1. 低ズームでは区単位サマリー、高ズームでは個別施設を描画する
**Rationale**: この地図でユーザーが把握したいのは件数の塊ではなく「どの区の情報か」である。ズームアウト時の主な負荷は「2,000件超の個別マーカーを同時に持つこと」にあるため、低ズームでは施設を区単位のサマリー表示へ切り替え、高ズームでのみ個別施設マーカーを描画する。これにより DOM・レイアウト負荷を抑えつつ、地図の意味論を維持できる。

**Alternative considered**: 件数ベースの一般的な地図クラスタを使う。性能改善は見込めるが、このプロジェクトでは区という行政区分が情報理解の単位であり、ドメインに合わないため不採用。

### 2. 個別施設マーカーはデザインを維持した画像資産として再利用する
**Rationale**: マーカーデザインは固定条件のため、単純な `CircleLayer` 置換は使えない。代わりに、カテゴリ組み合わせと表示スタイルごとに同一見た目の資産を生成・キャッシュし、MapLibre の描画に再利用する。これにより「1施設ごとの独立 SVG 構築」を避けながら見た目を維持できる。

**Alternative considered**: DOM `Marker` をそのまま使い続けて SVG 計算だけ軽くする。改善幅が限定的で、ズームアウト時のボトルネックを解消しにくいため不採用。

### 3. popup 表示は `selectedFacilityId` / `selectedFacility` に集約する
**Rationale**: popup は常に 1 件しか表示しないため、施設ループ内で popup を条件付き生成する必要はない。選択施設 state を単一化し、デスクトップは単一 `Popup`、モバイルは単一ボトムシートをその state から描画する。これにより責務が明確になり、描画戦略変更時にも popup ロジックを独立して維持できる。

**Alternative considered**: デスクトップとモバイルで別々の選択 state を持つ。分岐が増え、挙動差分の管理コストが上がるため不採用。

### 4. 施設データは一度ロードした結果をキャッシュし、地図表示用データを派生させる
**Rationale**: `static/recycling.db` のサイズと件数であれば、毎回のフィルタ変更で全件を再生成するより、元データをメモリに保持して派生データのみ更新する方が扱いやすい。検索、popup 解決、marker source 更新で同じ配列を使い回せるようにする。

**Alternative considered**: 現状どおり区・カテゴリ変更のたびに DB から再取得する。シンプルだが、今後の件数増加時に再計算コストが積み上がるため不採用。

### 5. マーカー描画戦略は専用モジュールへ分離する
**Rationale**: `+page.svelte` から、GeoJSON source 生成・区サマリー集約・マーカー資産解決・popup 選択解決のロジックを切り出す。画面は「state を渡して描画する」責務に寄せ、今後の調整を局所化する。

**Alternative considered**: 既存ファイル内で最適化を積み増す。短期的には可能だが、複雑さが増えやすく保守性が落ちるため不採用。

## Risks / Trade-offs

- **[Risk] 区サマリー表示により低ズーム時の見え方が変わる** → Mitigation: これは低ズーム帯に限定し、高ズームで既存マーカー表現へ確実に戻す
- **[Risk] 画像資産キャッシュのキー設計が不十分だと誤表示が起きる** → Mitigation: カテゴリ組み合わせ、marker style、solid color を含めた安定キーを使う
- **[Risk] popup の単一 state 化でモバイル・デスクトップ差分が壊れる** → Mitigation: 表示コンテンツを共通化し、表示コンテナだけを viewport で切り替える
- **[Risk] データキャッシュがフィルタ変更や設定変更と同期しない** → Mitigation: 元データ、表示データ、選択施設解決を段階的な derived state に分ける

## Migration Plan

1. `AGENTS.md` に `jj` bookmark 分離方針を追記する
2. 高速化用 `jj` bookmark を作成し、OpenSpec change を分離する
3. popup を単一選択 state へ集約する設計を artifact に反映する
4. 区単位サマリー表示とマーカー資産再利用の要件を spec 化する
5. 実装フェーズで popup 単一化から先に進め、その後に描画最適化を段階的に導入する
