## Context

現在の `SettingsSidebar.svelte` には `使い方` `データについて` `更新情報` `プライバシーポリシー` の4項目が並んでいるが、いずれも未接続で、利用可能かどうかが見た目から分かりにくい。既存の `sidebar-navigation` 変更で情報項目の優先順とリンク可能な構造は整理されているため、この change ではその最初の実体として `使い方` ページを追加し、利用者が現在の UI を自力で理解できる状態を作る。

この変更は UI 導線と静的コンテンツが中心で、データベースや GeoJSON の更新は含まない。そのため GeoJSON スキーマや施設データの重複排除ロジックには変更を加えず、利用ガイド本文でも掲載データの網羅性を誤認させない表現を採用する。

## Goals / Non-Goals

**Goals:**
- サイドバーの `使い方` をアプリ内で開ける実在ページに接続する
- 区の選択、カテゴリ絞り込み、マーカーの見方、表示設定を短く説明する
- 画面実装と乖離しない説明内容にして、今後 UI が変わった際も追従しやすい構成にする
- 未実装の情報項目と、利用可能な `使い方` の見え方を区別できるようにする

**Non-Goals:**
- `データについて` `更新情報` `プライバシーポリシー` の本文作成
- 新しいデータ取得、GeoJSON 編集、SQLite 生成物の変更
- チュートリアルのモーダル化や初回オンボーディングの追加
- サイドバー全体の情報設計を再度作り直すこと

## Decisions

### 1. `使い方` は専用ルートの静的ページとして実装する

`使い方` はサイドバー内で展開する説明パネルではなく、SvelteKit の専用ルートとして表示する。これにより URL で直接開ける、将来ほかの情報ページにも同じ実装パターンを使える、長めの説明でもレイアウトが崩れにくい、という利点がある。

代替案としてサイドバー内アコーディオンやモーダルも考えられるが、既存サイドバーは表示設定と情報導線を分ける方向で整理されており、読むための長文は独立ページの方が役割分離に合う。

### 2. サイドバーの情報項目は `href` の有無で利用可能状態を表現する

既存の `infoMenuItems` 構造を活かし、`使い方` のみ `href` を付与してリンク表示にする。未実装項目は現行どおりプレースホルダー扱いにしつつ、仕様上は「未利用可能」と分かる見え方を与える想定にする。

代替案として全項目を同時にダミーページへ接続する方法もあるが、内容の薄いページを量産するより、まず最優先項目だけを正しく出す方が情報品質を保ちやすい。

### 3. 利用ガイド本文は「現在存在する操作」だけを説明する

本文は以下の 4 つのセクションに絞る。
- 区の選択方法
- カテゴリ絞り込みの考え方
- 地図マーカーと施設確認の見方
- 表示設定の使い方

この構成なら現在の UI と対応づけやすく、仕様変更があっても差分更新しやすい。掲載区一覧、回収制度の一般論、外部制度説明のような周辺情報は含めない。

### 4. データ品質への配慮は説明文の制約として扱う

この change 自体はデータを変更しないが、利用ガイド内では「掲載データはアプリ内で対応している区・施設に限られる」「区ごとに受け入れカテゴリが異なる」といった前提を明示する。これにより、データの完全網羅や一律回収を誤認させるリスクを抑える。

「重複排除戦略」と「GeoJSON スキーマ要件」については、本変更では新しいデータ投入や変換を行わないため既存の仕組みを維持する。設計上も、利用ガイドの実装は GeoJSON の構造やデータ加工処理に依存しない静的コンテンツとして切り離す。

## Risks / Trade-offs

- [UI 変更と案内文がずれる] → 説明文は画面の正式名称に近い短い表現にとどめ、実装変更時に見直す対象としてまとめて管理する
- [未実装項目との見え方の差が弱い] → `使い方` を明確なリンク表示にし、未実装項目は押しても同じ挙動にならないよう仕様で分ける
- [情報ページが増えるとレイアウト方針が散らばる] → 今回の `使い方` ページを、今後の情報ページの共通パターンとして扱う

## Migration Plan

新規ページとサイドバー導線の追加のみで、データ移行は不要。問題があれば `使い方` の `href` を外してサイドバー上の未実装状態へ戻せる。
