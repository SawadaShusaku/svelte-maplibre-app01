## Context

現在の `svelte-maplibre-gl` の `<Popup>` コンポーネントは、地図上の特定の位置 (`lnglat`) に固定されており、ドラッグで移動する機能を提供していません。ユーザーがポップアップを動かせるようにするには、ドラッグ操作を検知し、ポップアップの位置を更新するロジックを実装する必要があります。

## Goals / Non-Goals

**Goals:**
- ポップアップを地図のコンテナ内でドラッグ可能にする。
- ドラッグ操作はデスクトップ（マウス）とモバイル（タッチ）の両方で機能する。
- 実装はパフォーマンスが良く、ドラッグ中に遅延が発生しないようにする。

**Non-Goals:**
- ポップアップのサイズ変更機能。
- ポップアップを閉じた後に位置を記憶する機能。
- この機能のために新しい外部ライブラリを導入すること。

## Decisions

### 決定事項: `svelte-maplibre-gl` の `<Popup>` の代わりに `maplibregl.Popup` を直接利用する

`svelte-maplibre-gl` の `<Popup>` コンポーネントは宣言的で便利ですが、ドラッグ機能のような低レベルの操作には向いていません。CSS `transform` を使って無理やり移動させると、地図のズームや移動時にポップアップの位置がずれる問題が発生します。

そこで、より堅牢なアプローチとして、`maplibre-gl` 本体が提供する `maplibregl.Popup` クラスを直接利用することに決定します。

1.  **`draggable` オプションの活用**:
    `maplibregl.Popup` はコンストラクタで `draggable: true` というオプションを受け付けます。これを有効にすることで、ライブラリ標準の安定したドラッグ機能を利用できます。

2.  **ポップアップのライフサイクル管理**:
    Svelte の `$effect` を使用して、`openPopupId` の変更を監視し、ポップアップのインスタンスを管理します。
    - `openPopupId` にIDが設定された場合:
        1.  現在表示されているポップアップがあれば削除する。
        2.  新しい `maplibregl.Popup({ draggable: true })` インスタンスを作成する。
        3.  ポップアップの中身となるDOM要素を生成し、`setDOMContent()` で設定する。
        4.  `setLngLat()` で位置を設定し、`addTo(map)` で地図に追加する。
    - `openPopupId` が `null` になった場合:
        - 表示されているポップアップを削除する。

3.  **ポップアップのコンテンツ**:
    ポップアップの中身はSvelteコンポーネントで作成します。しかし、`maplibregl.Popup` はDOM要素を要求するため、Svelteコンポーネントを直接渡すことはできません。
    この問題を解決するため、以下のアプローチを取ります。
    1.  ポップアップの中身だけを定義した新しいSvelteコンポーネント (`PopupContent.svelte`) を作成します。
    2.  `$effect` 内で、この `PopupContent` コンポーネントを `document.body` の外にあるコンテナ `div` にマウントします。
    3.  そのコンテナ `div` を `popup.setDOMContent()` に渡します。
    4.  ポップアップが閉じられる際には、Svelteコンポーネントを適切に破棄 (`$destroy`) します。

この方法により、`maplibre-gl` のネイティブなドラッグ機能を活用しつつ、ポップアップのコンテンツはSvelteで宣言的に記述できるという、両方のライブラリの長所を活かした実装が可能になります。

## Risks / Trade-offs

- **複雑性の増加**: `svelte-maplibre-gl` の宣言的な `<Popup>` を使う場合に比べ、手動でのインスタンス管理やコンポーネントのマウント・破棄が必要になるため、コードの複雑性が若干増します。しかし、これにより安定的で高機能なドラッグが実現できるため、トレードオフとして許容します。
- **Svelteコンポーネントの手動マウント**: Svelteコンポーネントを動的にマウント・破棄する部分は、注意深く実装しないとメモリリークの原因になる可能性があります。`$destroy` を確実に呼び出すことで、このリスクを軽減します。
