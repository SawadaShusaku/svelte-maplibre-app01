## Context

現在、リサイクル拠点データはGeoJSONファイル（`src/lib/data/{prefecture}/{city}.geojson`）を単一の情報源として管理している。ビルド時に `npm run build:db` でGeoJSONをSQLiteに変換し、アプリがブラウザ上で読み込む。しかし、GeoJSONデータに対する自動検証が存在しないため、以下の問題が発見された。

- **文字化け**：47件のFeatureの `name` または `address` に `` (U+FFFD) が混入しており、施設名が読めない
- **市区町村と住所の不整合**：`shinagawa.geojson` に「東京都稲城市長峰1丁目1番地」の施設が含まれている
- **不完全な住所**：52件のFeatureが「北区」「杉並区」など番地・建物名を欠いている
- **重複データ**：119組の同一住所Feature（うち多くは不完全住所が原因）

これらの問題は**人間が直接GeoJSONを修正する**ことで解決する。AIやスクリプトが勝手にデータを書き換えることは絶対に行わない。

## Goals / Non-Goals

**Goals:**
- 全GeoJSONファイルを対象とした自動データ品質監査スクリプトを作成する
- **文字化け検出**を首要な検証ルールとして追加する
- 修正が必要なデータを、人間が作業しやすい**一覧表形式（Markdown表/CSV）**で出力する
- ビルドパイプラインにデータ検証を統合し、**問題がある場合はビルドを失敗させ、人間に修正を促す**

**Non-Goals:**
- **自動データ修正**（スクリプトは検出のみ、書き換えは行わない）
- ブラウザ実行時のリアルタイムデータ検証（ビルド時のみ対象）
- 外部APIを用いた住所の正確性検証（GSIジオコーディングは別フローで対応）

## Decisions

### 1. 監査スクリプトはNode.js + 標準ライブラリで実装
**Rationale**: プロジェクトですでにNode.js/Viteが使用されているため追加依存が不要。`fs`, `path`, `glob` でGeoJSONを読み込み、軽量な検証ロジックを記述する。TypeScriptで記述し `tsx` または `vite-node` で実行する。

**Alternative considered**: Pythonスクリプト — プロジェクトにPythonランタイムがないため却下。

### 2. 検証は `npm run build:db` の前段階で実行
**Rationale**: SQLiteはビルドアーティファクトであり、汚染されたデータがDBに流れ込むことを防ぐため。`package.json` に `npm run audit:data` を新設し、`build:db` の `prebuild:db` として呼び出すか、または `build:db` スクリプト内で最初に実行する。

**Alternative considered**: ビルド後のDB検証 — GeoJSONを直接修正する方が情報源に近いため、GeoJSONレベルで検証する方が適切。

### 3. 自動修正は絶対に行わない
**Rationale**: GeoJSONファイルはデータの単一情報源であり、プログラムによる勝手な書き換えはデータの信頼性を損なう。監査スクリプトは**読み取り専用**とし、検出結果を人間に報告するのみとする。

**Concrete rule**: スクリプトに `fs.writeFileSync` や `JSON.stringify` によるファイル書き込み処理を含めない。

### 4. エラー発生時は非ゼロexitコードで終了し、修正データ一覧表を出力
**Rationale**: CI/CDでの利用を想定しつつ、人間が修正作業をする際の利便性も確保。一覧表はMarkdown形式で、人間がそのままTODOリストやスプレッドシートに貼り付けられる形式とする。

**Output format**:
```markdown
| ファイル | ID | 施設名 | 住所 | 問題 | 備考 |
|---|---|---|---|---|---|
| tokyo/shinagawa.geojson | shinagawa-022 | 総合体育館 | 東京都稲城市長峰1丁目1番地 | city不整合 | 品川区ファイルに稲城市住所 |
| tokyo/shinagawa.geojson | shinagawa-051 | 螢戰螢読霏⊂山店 | ... | 文字化け | nameに``混入 |
```

### 5. 住所重複検出は正規化後のファジーマッチング
**Rationale**: 完全一致では半角/全角の違いや空白の有無で重複を見逃す。住所を正規化（全角数字→半角、空白除去、「丁目/番地/号」の統一）してから比較する。

**Alternative considered**: 座標ベースの重複検出 — ジオコーディング誤差により同一施設でも座標がずれる場合があるため、住所ベースを主とし座標は補助とする。

### 6. 住所完全性チェックは日本語住所パターンに対応
**Rationale**: 「区名のみ」だけでなく、「丁目」までしかない住所も不完全。正規表現で「数字+丁目」「数字+番地」「数字+号」「○○町/○○通り」などのパターンを検出し、番地相当の情報が含まれているか確認する。

## Risks / Trade-offs

- **[Risk] 誤検出（False Positive）によりビルドが不安定化** → Mitigation: 重大度（error/warning）を分離し、warningはビルドを停止しない。最初の導入時は `--strict` フラグなしでwarningのみとし、運用安定後にerror化する。
- **[Risk] 大規模データセットでの実行時間** → Mitigation: 全ファイルの合計でも数MB程度であり、Node.jsでの同期読み込みでも十分高速。必要に応じて非同期I/Oに変更する。
- **[Risk] 文字化け検出が過剰に反応する** → Mitigation: `` のみを検出対象とし、異体字や旧字体など意図的な文字は対象外とする。
- **[Risk] 人間が修正一覧を見落とす** → Mitigation: コンソール出力に色付け（赤=error、黄=warning）し、最後に「=== 修正必要データ一覧 ===」として一覧表を再度出力する。

## Migration Plan

1. `scripts/audit-geojson.ts` を新規作成（読み取り専用）
2. `npm run audit:data` を `package.json` scriptsに追加
3. **人間が** 文字化け・不整合・不完全住所を含むGeoJSONファイルを修正
4. 修正後、`npm run audit:data` → `npm run build:db` → `npm run smoke` を実行して検証
5. 全問題が解決したら、通常運用に移行
