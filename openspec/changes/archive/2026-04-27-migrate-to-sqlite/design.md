## Context

現在のシステムはGeoJSONファイルを直接使用して回収拠点データを管理しています。データは以下の構造で保存されています：

```
src/lib/data/
├── tokyo/
│   ├── toshima.geojson
│   └── chiyoda.geojson
```

各GeoJSONはFeatureCollectionで、施設情報とカテゴリがpropertiesに含まれています。この構造では、以下の要件に対応するのが困難です：

1. **区ごとのカテゴリ管理**: 各区が対応するカテゴリが異なる
2. **回収団体の違い**: 小型充電式電池（JBRC）と乾電池（区施設）で回収先が異なる
3. **注意事項の管理**: 水銀対応、端子絶縁などカテゴリごとの注意事項
4. **動的フィルタリング**: 選択中の区に応じたUI表示の切り替え

## Goals / Non-Goals

**Goals:**

- リレーショナルデータベース（SQLite）を導入し、データの整合性を確保
- 区ごとのカテゴリ対応状況を管理できるようにする
- 回収団体情報をデータベース化し、検索・フィルタリングを可能にする
- 既存のUI/UXを維持しつつ、データ層のみを変更
- GeoJSONからの段階的な移行パスを提供

**Non-Goals:**

- 管理画面やCRUD APIの作成（将来的な拡張として別途検討）
- リアルタイム同期機能
- ユーザー認証・権限管理
- 既存のカテゴリ表示UIの大幅な変更

## Decisions

### 1. SQLiteを選択

**選択**: SQLiteを埋め込みデータベースとして使用

**理由**:
- 既存の静的ホスティング構成を維持可能
- ファイルベースでバックアップ・バージョン管理が容易
- クライアントサイドでの全文検索が可能
- 将来的なPostgreSQL移行も比較的容易

**代替案検討**:
- IndexedDB: 複雑なリレーションが書きにくい
- PostgreSQL: ホスティングコストと管理オーバーヘッド増

### 2. スキーマ設計（正規化）

**選択**: 7テーブルの正規化スキーマ

```
categories (カテゴリマスター)
category_details (カテゴリ詳細: 注意事項など)
collectors (回収団体)
wards (区・市情報)
ward_categories (区が対応するカテゴリ - 中間テーブル)
facilities (回収拠点)
facility_categories (施設が対応するカテゴリ - 中間テーブル)
```

**理由**:
- 多対多関係（区↔カテゴリ、施設↔カテゴリ）を適切に表現
- 回収団体の再利用（JBRCなど複数区で共通）
- 将来的な拡張性（営業時間テーブル追加など）

### 3. データ移行戦略

**選択**: ビルド時マイグレーションスクリプト

**理由**:
- デプロイ時に最新データを含められる
- 開発時と本番で同じデータ構造を保証
- GeoJSONとSQLiteの二重管理を回避

**実装案**:
```typescript
// scripts/migrate-to-sqlite.ts
// GeoJSONを読み込んでSQLiteを生成
// ビルド時に実行: npm run build:db
```

### 4. 段階的移行

**選択**: ダブルライト期間の設けない、即座の切り替え

**理由**:
- データ層のみの変更でUI変更がない
- 既存テストで動作検証可能
- ロールバックはgit revertで対応

## Risks / Trade-offs

**[リスク] ビルドサイズの増加**
→ SQLite WASMバイナリ（約500KB）が追加される
→ **軽減**: gzip圧縮で約200KBに削減可能。初期ロード後のキャッシュで影響を最小化

**[リスク] データ同期の複雑化**
→ 既存のGit管理GeoJSON → SQLite生成 → ビルドという流れが増える
→ **軽減**: 移行スクリプトを徹底的にテストし、CIで検証

**[リスク] ブラウザ互換性**
→ SafariのPrivateモードでIndexedDB制限がある
→ **軽減**: SQLiteはオンメモリまたはファイルシステムAPIを使用、基本的に互換性問題なし

**[リスク] デバッグ困難性**
→ バイナリSQLiteファイルはGit diffで確認できない
→ **軽減**: 開発時は常にSQLダンプを生成し、差分確認可能にする

## Migration Plan

### Phase 1: スキーマ作成 (Day 1)
1. SQLiteスキーマ定義（schema.sql）作成
2. TypeScript型定義の更新
3. マイグレーションスクリプト作成

### Phase 2: データ層実装 (Day 2-3)
1. `src/lib/db.ts` - SQLiteアクセス層作成
2. `src/lib/data.ts` - 既存インターフェースをSQLite対応に
3. ビルド設定更新（Viteプラグイン）

### Phase 3: データ移行 (Day 4)
1. 既存GeoJSONデータの移行
2. 23区データの確認と補完
3. 新カテゴリ（充電式電池分離など）の適用

### Phase 4: 検証・デプロイ (Day 5)
1. 既存機能の回帰テスト
2. パフォーマンス測定
3. ステージングデプロイ
4. 本番リリース

**Rollback Strategy:**
- Git revertでGeoJSON版に即座に戻せるように、コミットを分割
- SQLite版リリース後1週間はGeoJSONも残しておく（デュアルサポート）
